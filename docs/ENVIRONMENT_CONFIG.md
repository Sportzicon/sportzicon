# Environment Configuration Guide

This guide explains how to configure Sportivox for different environments: local development, staging, and production.

## Overview

Sportivox supports three distinct environments:

| Environment | Purpose | Database | Storage | Secrets |
|-------------|---------|----------|---------|---------|
| **Local** | Development on your machine | Firestore Emulator | GCS Emulator | `.env.local` files |
| **Staging** | Testing before production | Real GCP Firestore | Real GCS | GCP Secret Manager |
| **Production** | Live application | Real GCP Firestore | Real GCS | GCP Secret Manager |

## Environment Files

### Local Development (.env.local)

Files:
- `.env.local` — Global local config (root)
- `backend/.env.local` — Backend local config
- `frontend/.env.local` — Frontend local config

**Characteristics:**
- ✅ Relaxed rate limits (1000 requests, 50 auth attempts)
- ✅ Debug logging
- ✅ Firestore and GCS emulators (no GCP credentials needed)
- ✅ Weak JWT secrets (OK for dev)
- ✅ Local URLs (localhost:8080, localhost:5173)

**Setup:**
```bash
# Files already created - nothing needed
# Just run:
docker compose up --build

# Frontend will be at http://localhost:5173
# Backend API at http://localhost:8080
```

### Staging (.env.staging.example)

File: `.env.staging.example` — Copy and customize to `.env.staging`

**Characteristics:**
- 🔧 Standard rate limits (300 requests, 20 auth attempts)
- 🔧 Info logging
- 🔧 **Real GCP services** (Firestore Native, GCS buckets)
- 🔧 Strong JWT secrets from Secret Manager (or local override)
- 🔧 Staging URLs (api-staging.sportivox.com, staging.sportivox.com)

**Setup:**
```bash
# 1. Copy template
cp .env.staging.example .env.staging

# 2. Edit .env.staging with YOUR GCP project ID and secrets
# 3. Build and test locally:
docker compose -f docker-compose.prod.yml up --build

# 4. Push to GCP:
scripts/deploy-staging.sh
```

### Production (.env.production.example)

File: `.env.production.example` — Copy and customize to `.env.production`

**Characteristics:**
- 🔒 Strict rate limits (100 requests, 10 auth attempts)
- 🔒 Warning logging only
- 🔒 **Real GCP services** (Firestore Native, GCS buckets)
- 🔒 Strong JWT secrets generated by Terraform
- 🔒 Production URLs (api.sportivox.com, app.sportivox.com)

**Setup:**
```bash
# 1. Copy template
cp .env.production.example .env.production

# 2. Edit .env.production with YOUR production values
# 3. Configure Terraform variables (terraform/terraform.tfvars.prod)
# 4. Deploy:
scripts/deploy-prod.sh
```

## Key Configuration Differences

### Database & Storage

**Local:**
```
FIRESTORE_EMULATOR_HOST=firestore:8081
STORAGE_EMULATOR_HOST=http://gcs:4443
```

**Staging/Production:**
```
FIRESTORE_EMULATOR_HOST=  # EMPTY - use real Firestore
STORAGE_EMULATOR_HOST=    # EMPTY - use real GCS
GCP_PROJECT_ID=your-project-id
```

### Rate Limiting

Local dev (relaxed):
```
RATE_LIMIT_MAX=1000
AUTH_RATE_LIMIT_MAX=50
```

Staging (standard):
```
RATE_LIMIT_MAX=300
AUTH_RATE_LIMIT_MAX=20
```

Production (strict):
```
RATE_LIMIT_MAX=100
AUTH_RATE_LIMIT_MAX=10
```

### Logging

Local: `LOG_LEVEL=debug` — See everything  
Staging: `LOG_LEVEL=info` — Normal operation  
Production: `LOG_LEVEL=warn` — Errors and warnings only

### CORS Origins

Local:
```
CORS_ORIGINS=http://localhost:5173,http://localhost:3000
```

Staging:
```
CORS_ORIGINS=https://staging.sportivox.com
```

Production:
```
CORS_ORIGINS=https://app.sportivox.com
```

## Common Tasks

### Running Local Development

```bash
# Start all local services with emulators
docker compose up --build

# Frontend at http://localhost:5173
# Backend at http://localhost:8080
```

### Testing Production Build Locally

```bash
# Build and run with production settings (but still using emulators)
docker compose -f docker-compose.prod.yml up --build

# This tests:
# ✓ Production Docker images
# ✓ Production Node environment
# ✓ Production rate limits
# ✗ Real GCP services (still uses emulators)
```

### Checking Which Environment is Running

**Backend:**
```bash
# Check the API logs
docker compose logs api | grep "NODE_ENV"
```

**Frontend:**
```bash
# Check browser console
console.log(import.meta.env.VITE_API_BASE_URL)
```

### Rotating Secrets

**Local Development:**
Edit `.env.local` and restart:
```bash
docker compose down -v
docker compose up --build
```

**Production (GCP):**
Use GCP Secret Manager:
```bash
gcloud secrets versions add JWT_ACCESS_SECRET --data-file=secret.txt
# Redeploy Cloud Run to pick up new version
terraform apply
```

## Troubleshooting

### "Cannot connect to database" locally

Check if Firestore emulator is running:
```bash
curl http://localhost:8081  # Should return HTML, not error
```

If not running:
```bash
docker compose up firestore
```

### "Invalid JWT secret" in production

JWT secrets must be at least 64 characters. Generate strong random values:
```bash
openssl rand -hex 32  # 64 hex chars
```

### Accidentally deployed dev config to production

⚠️ **CRITICAL**: Immediately:
1. Stop the Cloud Run service
2. Rotate all secrets in Secret Manager
3. Redeploy with correct production config
4. Check audit logs for suspicious activity

**Prevention:**
- Never copy `.env.local` to production
- Always use `.env.production` for prod deploys
- Use separate GCP projects for dev/staging/prod

### CORS errors in staging/production

Verify `CORS_ORIGINS` matches your actual frontend URL:
```bash
# Check what's configured
grep CORS_ORIGINS .env.staging

# Update if needed
sed -i 's/CORS_ORIGINS=.*/CORS_ORIGINS=https://staging.yourdom.com/' .env.staging
```

## Best Practices

✅ **DO:**
- Keep `.env.*.local` files in gitignore (never commit secrets)
- Copy `.env.*.example` files and customize for your setup
- Use GCP Secret Manager for production secrets
- Test production configs locally before pushing
- Document your custom configuration choices

❌ **DON'T:**
- Commit `.env.production` to git
- Share secrets in Slack, email, or chat
- Use the same secrets for dev and production
- Hardcode secrets in code or Docker images
- Skip environment variable validation (Zod catches errors early)

## Configuration Checklist

### Before Local Development
- [ ] `.env.local` exists (or use defaults)
- [ ] Docker daemon is running
- [ ] Ports 5173, 8080, 8081, 4443 are free

### Before Staging Deployment
- [ ] `.env.staging` file created from template
- [ ] GCP Project ID is correct
- [ ] JWT secrets are strong (64+ chars)
- [ ] Staging URLs are set correctly
- [ ] `docker compose -f docker-compose.prod.yml up` works locally

### Before Production Deployment
- [ ] `.env.production` file created from template
- [ ] All secrets are strong and unique
- [ ] Production URLs are correct (app.sportivox.com)
- [ ] CORS_ORIGINS matches frontend domain
- [ ] terraform.tfvars is configured
- [ ] `terraform plan` shows expected changes
- [ ] Backup of current prod config is saved
- [ ] Team is notified of deployment
